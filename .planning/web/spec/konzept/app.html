§app2 Phases

§app2.1 Phase 1 - Web Readiness

Tool Purpose
Vite with web build target Produces a container without Electron modules.
Fastify HTTP backend; the port is read from ENV, a SIGTERM handler is wired in.
Authentication abstraction (Argon2 as the first implementation) Login for multiple users. The password hash stays with @node-rs/argon2 or a pure WASM variant inside the container.
Repository interface (StorageSession solidified) The loopholes are closed; the service classes talk exclusively against the interface.
Worker adapter worker_threads runs inside the container. Only cosmetic adjustments to paths and imports are needed.
Input and output adapters Native dialogs for Electron, plus multipart upload and download for the web variant.
Conditional build of native modules better-sqlite3-multiple-ciphers is built only on the Electron path.
Database connection SQLite by default (per Task Profile preference) or Postgres depending on the assessment in §bewertung2. The user_id columns receive default value 1.
Pino, /healthz, and migration tool Application guarantees from the Contract.

Bridge to Stage 2 (replaces the Stage 2 phases without having to break phase 1):

Setup in Stage 1 Connection to Stage 2
SQLite chosen. The repository interface encapsulates the database. The schema already contains the user_id columns and the audit_log table from migration version v7.
Authentication via Argon2. The authentication abstraction allows an OIDC implementation as a second variant.
Single-user login with user_id=1. Schema preparation for multiple users runs in parallel (§adr3).
Worker adapter. Remains unchanged in Stage 2.

Forbidden in phase 1 (would block the path to Stage 2):

  • Authentication directly in service code instead of behind an abstraction.
  • SQL strings specific to a particular DB engine without encapsulation in the repository interface.
  • Omitting the user_id columns, since this would turn migration to multi-user operation into a schema break.
  • Hard-coded file input and output without an adapter pattern.

Gate for phase 1:

Criterion
The web container starts without Electron dependencies.
Migrations run idempotently at startup.
/healthz correctly returns status codes 200 and 503.
Argon2 login works in the browser; multiple users can work in parallel.
Import, filtering, and analysis of variant data work in the browser. Core functionality is preserved.
The service classes access data exclusively through the repository interface.
The Electron variant continues to build without regression.
Logs are emitted as JSON to stdout.
SIGTERM terminates the process without data or request loss.
The entries §adr1, §adr2, and §adr3 are filed in the repository.
§bewertung1 (inventory) and §bewertung3 (Electron mapping) are up to date.
Bridge Clause check: the authentication abstraction is in place, the repository interface encapsulates the DB engine, and the user_id columns are set.
