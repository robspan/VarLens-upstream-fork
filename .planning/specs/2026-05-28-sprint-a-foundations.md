## Goal

Land the foundations that unblock 1000-genome scale: the PostgreSQL **cohort listing / page-load path** stops doing live `GROUP BY` over `variants` (specifically `PostgresCohortRepository.buildGroupedSelect` + `PostgresVariantReadRepository.getFilterOptions(caseId)`'s single-case fan-out to `getColumnMeta` — `getSummary`, `getGeneBurden`, and the multi-case `getColumnMeta({ caseIds })` branch are explicitly Sprint B per C4's scope note), the renderer stops paying N+1 + reactive-clone tax on every page-flip and case-switch, the PG read-side stops re-planning identical queries on every call, and a single `JobRunner` primitive replaces the four bespoke `inProgress` / progress-callback patterns spread across the import / association / batch surfaces. No partitioning, no JSONB, no concurrent imports — those are Sprint B / Sprint C.

This spec ships in **four independently shippable PRs** that the team can review and merge in any order PR-1 / PR-2 / PR-3, with PR-4 last. PR-1 + PR-2 are user-visible perf wins and tag as 0.66 / 0.67. PR-3 hits the audit's named exit criterion and tags as 0.67. PR-4 is foundation-with-no-behaviour-change and tags as 0.68 — the sprint exit tag.

| Sub-item | Audit ref | Summary |

| **B1: `runNamed` + `runNamedDynamic` helpers** | §3.6 | `src/main/storage/postgres/named-query.ts` — two wrappers. **(a) `runNamed<T>(pool, spec: { name; text; values; schema })`** for **static-SQL** call sites (the SQL text is a constant template; only the parameter values vary). **(b) `runNamedDynamic<T>(pool, spec: { baseName; text; values; schema })` (Pass-8 #3)** for **dynamic-SQL** call sites (the SQL text itself varies across calls, e.g. `queryVariants` whose WHERE clauses are conditionally appended via `buildQueryParts`). `runNamedDynamic` computes the effective name as `${baseName}:t${sha1(text).slice(0,8)}@${schemaToken(schema)}`. **Why this matters:** node-postgres checks `connection.parsedStatements[name]` at `pg/lib/query.js:156` and throws `Prepared statements must be unique` if the SAME name is registered with DIFFERENT text on the SAME connection — exactly what would happen if a single name like `variant:query_page:v1` was used for `queryVariants` calls with different filter shapes. The shape-hash tail makes each distinct text get its own prepared statement; repeat shapes share their cache slot per connection. **Memory caveat (Pass-9 #2):** prepared statements never release per-connection; high-cardinality dynamic shapes could accumulate. The cap CANNOT be implemented as "per-connection" through `pool.query` because that path checks out a transparent client — the wrapper doesn't see which client is selected. **Use a process-level effective-name cap instead:** maintain a process-global `Set<string>` of all effective names ever issued via `runNamedDynamic`; when the set's size exceeds `Math.max(64, 16 * top20Size)`, new dynamic-shape calls fall back to unnamed `pool.query(text, values)` and log once at WARN with the effective-name set size. The named-counter helper from B3 still increments named when applicable, so coverage drops are visible. (Alternative: have `runNamedDynamic` explicitly `pool.connect()` + `client.query` + `client.release()` to gain per-client visibility — rejected for PR-2 because it adds connection-checkout overhead to every dynamic-SQL call and complicates the read-replica future.) Sprint C revisits if telemetry shows churn. **Schema isolation (Codex Pass-1 #3 + Pass-2 verdict #2):** the effective prepared-statement name is `${spec.name}@${schemaToken(spec.schema)}` — e.g. `variants:query:v1@varlens_demo`. Current PG repositories interpolate schema names into SQL via the `"__schema__"."<table>"` placeholder pattern (visible across `0001_create_cases.sql` and friends); the SAME logical query against two schemas has DIFFERENT text after interpolation. Without per-schema name isolation, a pool connection that has prepared `variants:query:v1` against schema A would error or silently mis-resolve when re-used against schema B. The `@${schema}` suffix is mechanical, cheap, and aligns the prepared-statement cache key with the SQL text key. **Reserve `@` from logical names + token the schema:** logical `name` segments (the `<domain>:<purpose>:vN` portion) must not contain `@`; an agent-check grep enforces this. The schema portion is fed through `schemaToken(schema)` defined as `${slug}_${hash6}` where `slug = schema.toLowerCase().replace(/[^a-z0-9_]/g, '_').slice(0, 24)` and `hash6 = sha1(schema).slice(0, 6)`. **The hash tail is ALWAYS appended, not only on collision risk (Codex Pass-3 MED #4)** — otherwise `Case Lab`, `case-lab`, and `case_lab` would slug to the same `case_lab` and collide. ## Project-rule constraints (all PRs)

- **Branch discipline.** `AGENTS.md` forbids feature/work commits on `main`. Use a git worktree if the current checkout must stay clean — recommended for running multiple PRs in parallel.
- **Cohort parity** (`feedback_cohort_parity.md`). Every PR ships cohort-view parity in the same PR: PR-1 FilterToolbar deferred mount covers Case + Cohort views; PR-2 names cohort-view filter-preset reads in the same wave; PR-3 ships the storage-layer parity test; PR-4 has no cohort-specific surface.
- **Never lower thresholds** (`feedback_never_lower_thresholds.md`). Coverage / lint / typecheck thresholds stay where they are; failing suites are fixed by adding tests or fixing code.
- **Structured logging only.** New code uses `mainLogger` (main) or `logService` (renderer); never `console.*`.
- **Conventional Commits.** Allowed types: `feat`, `fix`, `refactor`, `perf`, `test`, `docs`, `style`, `chore`, `ci`, `merge`. Scope optional but encouraged.
- **LLM-sustainable source.** `make agent-check` green on every PR. New files target 150-400 lines; >600 lines requires PR-level justification in the description.
- **IPC contract.** Any change that touches IPC must keep `tests/shared/types/preload-contract.test.ts` green; new domains (PR-4 `jobs:`) extend the contract.
