> **Status (updated 2026-04-22):** Partially landed and retained as reference. Shipped from this spec: arm64-only macOS declaration, coverage-threshold restoration via added tests, scoring-via-filters vision doc, and Node-24 workflow action bumps. **Date:** 2026-04-11
**Status:** Partially landed, retained as historical design reference
**Scope:** Five connected loose ends after the v0.56.0 unified-shortlist release
**Session trigger:** Commit `5dc2beb` lowered two coverage thresholds to unbreak main-branch CI instead of writing the missing tests — a violation of the `feedback_never_lower_thresholds.md` rule. This spec fixes that and bundles four other post-release items that were discovered in the same session.

---

## 1 · Goals

1. **Undo the `5dc2beb` threshold lowering by adding tests**, not by reverting. The memory rule is not negotiable: quality gates only have value if they're defended.
2. **Formalize Apple-Silicon-only macOS support.** Make the arm64-only build an intentional declaration instead of a silent default dependent on the GitHub Actions runner's host architecture.
3. **Clear the two open Dependabot alerts** (vite + esbuild, both dev-time, both medium severity) without doing a Vite 8 major upgrade.
4. **Capture the scoring-via-filters redesign vision** as a planning-doc reference so future work on the three "Phase-1 scoring gaps" (inheritance forwarding, SV/CNV/STR rarity, HPO phenotype) has a coherent north star instead of drifting into three disconnected patches.
5. **Upgrade the deprecated GitHub Actions** that GitHub will force onto Node 24 by June 2026, before the forced cutover breaks CI.

## 2 · Non-goals

- Releasing a `v0.56.1` hotfix. No release cycle this session.
- Implementing any of the scoring-via-filters redesign. Item 4 is a vision document only; no code in the scoring module changes, no filter-builder refactors, no preset-format migration.
- Bumping Vite to 8.x. Vite 7 → Vite 8 is a major-version upgrade with breaking changes; it gets its own spec when its turn comes. If `npm audit fix` demands Vite 8 to clear the CVEs, item 3 is abandoned this session and refiled as a Vite-8 upgrade spec.
- Designing a replacement for `gitleaks/gitleaks-action@v2`. There is no v3 upstream. We stay on v2 this session and document the risk.
- Restoring the `inheritanceModes` field to the Recessive shortlist preset. That lands with its own spec once the scoring-via-filters redesign gives it a proper home (inheritance as a first-class filter predicate, not a `VariantFilterBuilder.build()` special case).

## 3 · Context

### 3.1 The offending commit (`5dc2beb`)

`vitest.config.ts` thresholds lowered:
- `src/main/services/scoring/**`: 95 / 90 / 95 / 95 → **92 / 90 / 95 / 90**
- `src/renderer/src/components/shortlist/**`: 75 / 65 / 75 / 75 → **70 / 60 / 70 / 70**

The commit message rationalized the lowering with "current coverage sits at ~93% / ~74%, thresholds chosen to leave a small safety margin below current numbers". That reasoning violates `feedback_never_lower_thresholds.md`: *"when coverage/lint/test thresholds fail, add tests or fix the code — lowering the threshold is a cover-up, not a fix."*

### 3.2 Why the coverage dropped in v0.56.0

Two distinct causes:

1. **`src/main/services/scoring/scoring-config.ts` extraction.** Wave 7 of the v0.56.0 spec extracted every numeric scoring constant into a single typed object literal. V8's line-coverage instrumentation records each field declaration as an executable statement, but none of the existing scorer tests assert the config values directly — they exercise them indirectly through compiled scorer outputs. Result: the config file's declaration lines stay grey.

2. **Shortlist Vue SFC render paths.** The three SFCs (`ShortlistPanel.vue`, `ShortlistTable.vue`, `RankScoreTooltip.vue`) have ~74% line coverage because their template event handlers (hover, dismiss, refresh, toggle-star broadcast, open-in-tab, v-list actions menu) and null/empty branches are not exercised by the existing unit tests. They *are* covered by the Playwright monkey test and the VitePress screenshot E2E run, but those are out-of-scope for the Vitest `--coverage` collector.

### 3.3 Open Dependabot alerts

| # | GHSA | Package | Severity | Summary |
|---|---|---|---|---|
| 40 | GHSA-4w7w-66w2-5vf9 | `vite` | medium | Path traversal in optimized-deps `.map` handling |

Both are exploitable only when a developer is running the Vite dev server AND visits a hostile website. They do not affect the shipped Electron binary. Low practical blast radius, but real alerts — "defend the quality gates" applies to CVE banners as much as coverage thresholds.

Current tree: `vite@7.3.2`, `esbuild` at three versions (`0.21.5` from vitepress → vite@5.4.21, `0.25.12` from electron-vite, `0.27.7` from vite@7.3.2). Latest: `vite@8.0.8`, `esbuild@0.28.0`.

### 3.4 Missing macOS x64 dmg

`package.json` `mac.target` is `["dmg", "zip"]` with no `arch` field. `electron-builder` on the `macos-latest` GitHub Actions runner (arm64 since 2024) defaults to host-arch-only builds, so v0.56.0 shipped with `Varlens-0.56.0-arm64.dmg` only — no x64 dmg. User decision: Intel Mac support is explicitly dropped. This spec formalizes that decision in config and documentation so it stops looking like an accidental regression.

### 3.5 Deprecated GitHub Actions (Node-20 runtime)

GitHub will force Node-20 actions to Node 24 by June 2026. Three actions currently pinned in our workflows are affected:

| Action | Current | Latest | Node 24 support | Action required |
|---|---|---|---|---|
| `actions/cache` | `@v4` | `v5.0.4` (2025-03-18) | v5+ only | Bump `@v4` → `@v5` |
| `actions/upload-artifact` | `@v4` | `v7.0.1` | v5+ | Bump `@v4` → `@v5` (conservative — v6/v7 add features we don't need) |
| `gitleaks/gitleaks-action` | `@v2` | `v2.3.9` (no v3) | Unclear | Stay on `@v2`, document risk |

`actions/checkout` and `actions/setup-node` are already on `@v6` (modern runtime). `.github/workflows/{build,release,docs}.yml` are affected.

### 3.6 The scoring-via-filters pivot (item 4 redesign)

Original item 4 was three separate "Phase-1 gaps" to close: SV/CNV/STR rarity wiring, `inheritanceModes` forwarding to Stage-1 shortlist query, HPO phenotype pipeline. During brainstorming the user pivoted: all three share a single root cause, which is that **the scoring module today reads raw DB columns (`row.rarity`, `row.consequence`) through a fixed formula, instead of composing filter-predicate matches**. Fixing the gaps by patching the current formula would ship three disconnected workarounds. Fixing them via a filter-first rewrite fixes all three at once AND makes future predicates (structural constraints, HPO similarity, allele-context boosts) trivial to add.

The redesign is too big to design and implement in this session. Item 4 is reshaped as a **vision document** that captures the new mental model and filter-audit findings so future subsystem specs have a shared reference point. The existing v0.56.0 scoring module is not deprecated by this doc — it remains in `main` until its replacement lands.

## 5 · Design — per item

### 5.1 Item 1 — Explicit arm64-only macOS

**Change in `package.json` `build.mac`** (line 111-118):
```json
"mac": {
  "icon": "build/icon.icns",
  "target": [
    { "target": "dmg", "arch": ["arm64"] },
    { "target": "zip", "arch": ["arm64"] }
  ],
  "artifactName": "${productName}-${version}-${arch}.${ext}"
}
```

Rationale: `electron-builder` accepts target entries as either string names or `{ target, arch }` objects. The object form is the only way to declare arch explicitly. Today we depend on the runner's host architecture, which is fragile — if GitHub ever changes the default runner architecture again, our build silently flips.

**Docs change:** Two files get concrete edits (confirmed during spec self-review):
- `README.md:49` — update the macOS install row from `| macOS | [DMG / ZIP](...) |` to `| macOS (Apple Silicon only) | [DMG / ZIP](...) |` so the install table itself reflects the constraint.
- `docs/guide/installation.md:30` — replace the existing line `- **macOS:** macOS 12 (Monterey) or later` with `- **macOS:** macOS 12 (Monterey) or later on Apple Silicon (M1 or newer). Intel Macs are not supported.` The file already has a `## System Requirements` section at line 27; this is the authoritative system-requirements page (README:52 links to it), and the table at lines 13-14 already references `arm64`-only assets, so only line 30 needs to change.

**CHANGELOG:** No entry this session. The note lands in the next release's changelog as a "Compatibility" bullet.

**Commit:** `chore(build): explicit arm64-only macOS target + docs` — inline on `main`.

### 5.4 Item 4 — Scoring-via-filters vision doc

**File:** `.planning/docs/2026-04-11-scoring-via-filters-vision.md`
**Length:** ~500–700 words
**Commit:** `docs(planning): scoring-via-filters vision` — inline on `main`

**Required sections:**

1. **Context** — one paragraph: v0.56.0 shipped the shortlist with a formula-based scorer, three documented Phase-1 gaps (inheritance forwarding, SV/CNV/STR rarity, HPO phenotype) all trace to a single architectural root cause.

2. **Mental model** — two paragraphs. Today: scoring is `sum(weights[dimension] * extractFeature(row, dimension))`. Tomorrow: scoring is `sum(weights[filterName] * filterMatches(variant, filterName))`. The unit of scoring becomes a **named filter predicate** that any variant type can be tested against. The preset carries a list of `{ filter, weight }` pairs rather than a fixed 5-dimensional weight vector.

3. **Worked example** — verbatim from the design section above ("Severe recessive" preset with three filter-weight rows, three variants scored against it). This makes the abstraction concrete for readers who skim.

4. **Filter audit** — the table from the design section, showing for each predicate family (gene symbol, rarity, inheritance, type-specific columns, FTS) which of SNV/SV/CNV/STR currently support it and what the blocker is. Ends with the "three root causes" bullets (inheritance is imperative SQL, shortlist-query doesn't share it, preset shapes diverge).

5. **Migration path (high-level)** — five bullets, no file names: lift inheritance into the filter registry, wire per-type rarity as filter predicates, unify preset JSON shapes, rewrite scoring module as weighted filter-match sum, (later) HPO phenotype as a filter predicate.

6. **Explicitly out of scope** — algorithm choices (Resnik vs simGIC), data-source decisions (gnomAD-SV bundling), implementation file lists. Each root cause gets its own spec when its turn comes.

7. **Not a deprecation of v0.56.0 scoring** — one sentence: the current formula-based scorers stay in `main` until their replacement lands.

**Source for the filter-audit section:** an Agent-tool audit already run during this session's brainstorming phase. Key findings the doc must accurately reflect:
- Inheritance mode lives in `VariantFilterBuilder.ts:561-694`, requires `analysis_group_id`, silently skipped when null
- `rarityPlaceholder: 1.0` hardcoded at `scoring-config.ts:163` (sv), `:224` (cnv), `:281` (str)
- Shortlist preset shape is `baseFilters` + `perTypeOverrides` (ShortlistConfig) vs classic filter preset's flat `FilterState` shape — these differ structurally
- SV/CNV/STR extension tables have rich type-specific columns that aren't wired into filter predicates yet
- CNV is the only extension table without FTS5 search (`hasFts: false`)

The doc must NOT contain code examples of proposed APIs — it's a direction document, not a design spec.
