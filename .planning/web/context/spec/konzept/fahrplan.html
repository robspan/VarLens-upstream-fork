Roadmap

What the Concept is and what it is not. The baseline delivers the runnable Concept: VarLens as a web application on a server, usable for the domain with test data and fake data. It is not production operations. Tools that would carry real operations - audit-trail hardening, off-host logging, append-only backups, OIDC federation, security hardening - are deliberately not yet attached. These building blocks live in ../betrieb/.

Consequence: no real data in the Concept. The required security and traceability building blocks are missing. Anyone wanting real data either runs the full Stage 2 playbook end to end, or selectively pulls forward building blocks from ../betrieb/ (see "Selective pulled-forward building blocks" at the end of this plan).

§fahrplan7 Selective pulled-forward building blocks - "minimal but production-grade"

If real-data ingestion becomes necessary before Stage 2 is fully completed, individual Stage 2 building blocks can be pulled forward. Possible candidates:

Pulled-forward building block Source in operations Effort Effect
OIDC integration ../betrieb/app.html §app2.2 (partial) medium Federation and MFA via central identity provider
Append-only audit ../betrieb/app.html §app2.4 medium Tamper protection on the audit table
Off-host logging ../betrieb/infrastruktur.html §infrastruktur3.4 (partial) small Protection against trace deletion
Object Lock backups ../betrieb/infrastruktur.html §infrastruktur3.3 (partial) small Protection against backup ransomware
WAF / fail2ban ../betrieb/infrastruktur.html §infrastruktur3.4 (partial) small Brute-force protection

Which building blocks are actually needed depends on stakeholder security requirements (see bewertungen.html §bewertung5). Pulling forward is a deliberate stakeholder decision, not a default.

Already pulled-forward building blocks

The following Stage 2 building blocks have been deliberately pulled forward into the Concept state to increase developer velocity. They do not replace the full Stage 2 stack (Netdata with metrics history, Diun for image-update watching, mail alert path, off-host persistence).

Pulled-forward building block Stage 2 reference Concept variant What remains open for Stage 2
Reverse proxy with TLS termination ../betrieb/infrastruktur.html §infrastruktur3.2 (Caddy) Caddy as a Compose service on port 80, without TLS until a domain is set Let's Encrypt auto-TLS once a domain is in place; routing to the application
Service status view (what is running, what is green) ../betrieb/infrastruktur.html §infrastruktur3.2 (Netdata) Uptime Kuma as a Compose service, behind Caddy with basic auth, only service pings and HTTP checks Metrics history with Netdata, mail alerts instead of manual configuration, multi-server view
Live container logs not explicitly part of the Stage 2 plan Dozzle as a Compose service, behind Caddy with basic auth, no persistence Persistent log aggregation per §infrastruktur3.4 (off-host logs)

Rationale: status and log views significantly increase developer velocity without producing meaningful platform lock-in or Bridge Clause violations. Both tools run as containers, are configured via ENV, and can be replaced without difficulty by Netdata or a fully featured off-host logging solution once Stage 2 is reached.
