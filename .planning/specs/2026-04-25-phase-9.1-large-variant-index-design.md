# PostgreSQL Parity Phase 9.1: Large Variant Coordinate Index

**Date:** 2026-04-25
**Status:** Proposed
**Depends on:**

- [PostgreSQL Parity Phase 7: Variants Read Parity](../archive/completed-specs/2026-04-24-postgresql-parity-phase-7-variants-read-parity.md)
- [PostgreSQL Parity Phase 9: VCF Import and PostgreSQL Import Worker](2026-04-25-postgresql-parity-phase-9-vcf-import-and-import-worker.md)

**Goal:** Unblock the postgres WGS perf gate (PR #179) by replacing the postgres coordinate-keyed btree indexes — `idx_variants_coord_case` on `variants(chr, pos, ref, alt, case_id)` and `variant_frequency.PRIMARY KEY (chr, pos, ref, alt)` — with sha256-keyed indexes that scale to arbitrary allele length, including the multi-KB HLA mega-alleles in GIAB HG002 and the multi-KB-to-multi-MB repeat expansions and indels common in long-read sequencing data.

## Background and Motivation

PostgreSQL btree indexes cap each index entry at one third of the page size, ~2704 bytes on the 8 KB default. Phase 6 declared `idx_variants_coord_case` over `(chr, pos, ref, alt, case_id)` to support cross-case coordinate lookups (trio inheritance, `getCarriers`, the `variant_frequency` JOIN). Phase 7 introduced `variant_frequency` with `PRIMARY KEY (chr, pos, ref, alt)`, which postgres backs with an implicit btree under the same limit.

Five GIAB HG002 v4.2.1 high-confidence variants on chr6 in the HLA region carry ref/alt strings of 2891–9705 bytes, which produce btree entries of ~3600–10500 bytes after per-tuple overhead. The first such row aborts the WGS import:

```
[PostgresImportExecutor] worker error:
  index row size 3600 exceeds btree version 4 maximum 2704 for index "idx_variants_coord_case"
```

SQLite has no per-row btree size cap and imports the same fixture in ~52 s, so the failure is postgres-specific and breaks the Phase 9 perf gate.

Long-read sequencing makes this worse, not rarer. Nanopore and PacBio variant callers materialize the explicit ref/alt for repeat expansions, large indels, and structural variants (rather than encoding them as symbolic alleles like `<DEL>`). It is normal to see ref or alt in the tens-of-KB range, and not unusual for repeat expansions to be hundreds of KB. Any approach that drops, truncates, or partially-indexes large alleles is wrong for that workload — it would silently lose data clinical and research users actually care about.

The fix has to scale to arbitrary allele length without losing rows, without requiring app-side allele truncation, and without forcing every cross-case query to fall back to seq-scan.

## Goals

1. The postgres WGS perf benchmark (`tests/perf/postgres-vcf-wgs-import.perf.test.ts`) imports the GIAB HG002 fixture cleanly with `VARLENS_RUN_WGS_PERF=1`.
2. The postgres `variants` table accepts INSERTs with ref/alt of arbitrary size — no schema-level truncation, no app-side rejection of large alleles.
3. Cross-case coordinate lookups (`variant_frequency` JOIN, `getCarriers` if/when ported to postgres, future trio-inheritance queries) use a btree index lookup that is `O(log N)` regardless of ref/alt length.
4. `variant_frequency` upserts (`ON CONFLICT … DO UPDATE`) work for any variant, including those with multi-KB ref/alt.
5. SQLite parity tests continue to pass with no SQLite schema change.
6. The full WGS comparison harness (`scripts/perf/compare-wgs-import.mjs`) runs to completion against both backends and writes a comparison artifact to `.planning/artifacts/perf/wgs-import/`.

## Non-Goals

- **No SQLite schema change.** SQLite has no btree row-size cap; the existing `idx_variants_coord_case` and `variant_frequency.PRIMARY KEY` work for any allele size SQLite can store. Mirroring the hash machinery into SQLite would cost ~330 MB of heap on a typical 10⁷-variant database for zero performance benefit. Existing user databases are not migrated. Repository-level divergence (postgres uses `coord_hash`, SQLite uses `(chr, pos, ref, alt)`) is intentional and consistent with existing backend asymmetry (`cohort_variant_summary` is also SQLite-only).
- **No interop with GA4GH VRS variant IDs.** VRS uses SHA512-truncated, base64url-encoded canonical IDs after a normalization pipeline (left-shift, etc.). VarLens's `coord_hash` is an internal lookup key over the *raw* `(chr, pos, ref, alt)` tuple — not a normalized canonical variant identifier. We pick sha256 here because it's the modern crypto default and built into postgres via `pgcrypto`; aligning with VRS is a separate, larger initiative that would also have to address allele normalization and reference assembly handling.
- **No replacement of the VCF symbolic-allele path.** Variants imported as `<DEL>`, `<INS>`, etc. continue to live in `variants.alt` as the symbolic string and in `variant_sv` for the structural metadata. The hash index doesn't change that.
- **No COPY FROM STDIN escalation.** That remains a Phase 16 polish item gated on the post-fix PG vs SQLite WGS comparison.
- **No collision-defense beyond the hash itself.** sha256 collision probability at our scale (~10⁹ variants ever → birthday-bound below 10⁻²⁰) is so far below all other failure modes — and column-equality "defense-in-depth" was *withdrawn* from the design after spec review noted that comparing detoasted multi-MB ref/alt values on every cross-case JOIN row would be a real perf hazard for long-read data. We rely on the cryptographic hash + the injective length-prefixed encoding (Decision 2) to prevent collisions, not on application-side fallback predicates.

## Architectural Decisions (Locked)

1. **PostgreSQL only.** SQLite is unaffected. Schema changes apply exclusively to `scripts/postgres/init-db/12-phase7-variants.sql` and the postgres app-side import/read repos. SQLite migrations and queries are untouched.

2. **sha256 via `pgcrypto`, with length-prefixed binary encoding.** The hash input is *not* a colon-delimited string — colon-delimited concatenation is not injective for VCF data (breakend ALTs like `G]chr2:1234]` contain colons, and a contig name `'1:2'` with `pos=3` collides on `'1:2:3:A:T'` with a contig `'1'`, `pos=2`, `ref='3:A'`). The hash input is a length-prefixed `bytea` blob:

   ```sql
   digest(
     int4send(octet_length(convert_to(chr, 'UTF8'))) || convert_to(chr, 'UTF8') ||
     int8send(pos) ||
     int4send(octet_length(convert_to(ref, 'UTF8'))) || convert_to(ref, 'UTF8') ||
     int4send(octet_length(convert_to(alt, 'UTF8'))) || convert_to(alt, 'UTF8'),
     'sha256'
   )
   ```

   Each variable-length field is prefixed by its byte length as a 4-byte big-endian integer (`int4send`); `pos` is encoded as an 8-byte big-endian `int8send`. This is unambiguously injective: the original tuple is recoverable from the byte stream by reading lengths and slicing. `convert_to(text, 'UTF8')` is explicit so the encoding doesn't drift if the database default encoding changes. `pgcrypto` is enabled with `CREATE EXTENSION IF NOT EXISTS pgcrypto;` in the init-db scripts. Md5 was rejected as deprecated-feeling in security contexts; SHA512 was rejected as 2× the storage cost without a corresponding interop benefit (since we are not aligning with GA4GH VRS in this phase).

   **Implementation note (PostgreSQL 18):** `convert_to(text, name)` is `STABLE`, and PostgreSQL 18 enforces `IMMUTABLE`-only inside `GENERATED ALWAYS AS … STORED`. The shipped expression in `12-phase7-variants.sql` therefore substitutes `col::bytea` for `convert_to(col, 'UTF8')`. The cast returns the raw server-encoding bytes; VarLens always runs postgres with `server_encoding = UTF8` (the docker-compose default and what CI uses), which makes the two forms byte-identical. The JS reference helper in `tests/main/storage/coord-hash-encoding.ts` continues to use fixed `Buffer.from(text, 'utf8')` — equivalence is verified end-to-end by the Task 7 E2E test, which compares postgres-stored `coord_hash` bytes against the JS-computed value for ASCII, multi-byte UTF-8, and 9.7 KB allele fixtures.

3. **Stored generated column, not expression-only index.** Both `variants` and `variant_frequency` gain a `coord_hash BYTEA GENERATED ALWAYS AS (digest(...)) STORED` column. The btree indexes target `coord_hash`, not the expression. Rationale: an expression-only index forces every consumer of the index to repeat the precise `digest(...)` argument list — a single character of drift makes the planner silently fall back to seq-scan, with no test that would catch it. A real column gives Kysely a typed handle and makes the query SQL `WHERE coord_hash = $1` instead of repeating the digest expression.

4. **Hash-only equality on read paths.** Cross-case JOINs and lookups use only `vf.coord_hash = v.coord_hash` (and the symmetric form for parameter lookups). The earlier draft of this spec proposed a "defense-in-depth" `chr = ? AND pos = ? AND ref = ? AND alt = ?` predicate alongside the hash equality; that was withdrawn after spec review noted that those equality predicates would force postgres to detoast and byte-compare the full `ref`/`alt` values, which for long-read mega-alleles can be megabytes per row. We accept the (negligible) sha256 collision probability instead. At our cardinality (≤ 10⁹ variants ever in a single VarLens database) the birthday-bound collision probability is below 10⁻²⁰ — well past cosmic-ray and CRC-stripped-disk-corruption rates. The injective length-prefixed encoding (Decision 2) means the hash function itself is correct; collisions can only come from the cryptographic hash, not from input ambiguity. If we ever observe a real collision in production we will reopen this decision.

5. **Surrogate primary key on `variant_frequency`.** Drop `PRIMARY KEY (chr, pos, ref, alt)`; add `id BIGSERIAL PRIMARY KEY`; add `UNIQUE INDEX variant_frequency_coord_hash_uniq ON variant_frequency(coord_hash)`. The upsert's `ON CONFLICT (coord_hash) DO UPDATE` targets the unique index. The `chr/pos/ref/alt` columns remain on the table and remain populated; only the constraint shape moves.

6. **Hash is computed only at the postgres layer, never client-side.** The renderer and main process never recompute the hash; postgres's `GENERATED ALWAYS AS … STORED` evaluates the digest at INSERT time, and parameter-side hash computation in cross-case lookups happens inline as part of the prepared SQL string. App-side drift between the column DDL and the lookup SQL is structurally impossible because the lookup SQL is the only other place the digest expression appears, and both live in the same git checkout. The unit test asserts equivalence between a JS-side reference implementation of the encoding (used to construct expected `coord_hash` values for fixtures) and what postgres actually stores; any drift fails CI loudly.

7. **`scripts/postgres/init-db/12-phase7-variants.sql` is edited in place.** VarLens postgres has no incremental migration framework yet (it's docker-entrypoint init scripts only). The canonical schema for fresh `pg-reset` containers lives in this file; that's where the change goes. Existing postgres dev databases require `make pg-reset && make pg-up` to pick up the change. CI's PG E2E always runs against fresh containers, so CI is unaffected. The PR description and PR comment will spell out the `pg-reset` requirement for reviewers.

8. **No data migration script.** Nobody is shipping VarLens with a pre-existing postgres database in production yet (postgres mode is still experimental, gated by `VARLENS_EXPERIMENTAL_STORAGE_BACKEND=postgres`). For dev/CI/test, `pg-reset` is the migration. If we ship postgres GA later, that ship gate will include migration tooling.

## Architecture

### Schema changes (`scripts/postgres/init-db/12-phase7-variants.sql`)

```sql
-- New: enable pgcrypto for digest()
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- variants table — add generated coord_hash column
CREATE TABLE IF NOT EXISTS variants (
  id BIGSERIAL PRIMARY KEY,
  case_id BIGINT NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  chr TEXT NOT NULL,
  pos BIGINT NOT NULL,
  ref TEXT NOT NULL,
  alt TEXT NOT NULL,
  -- … all existing columns unchanged …
  coord_hash BYTEA GENERATED ALWAYS AS (
    digest(
      int4send(octet_length(convert_to(chr, 'UTF8'))) || convert_to(chr, 'UTF8') ||
      int8send(pos) ||
      int4send(octet_length(convert_to(ref, 'UTF8'))) || convert_to(ref, 'UTF8') ||
      int4send(octet_length(convert_to(alt, 'UTF8'))) || convert_to(alt, 'UTF8'),
      'sha256'
    )
  ) STORED,
  search_document tsvector
);

-- variant_frequency: surrogate PK, drop coordinate PK, add hash unique
CREATE TABLE IF NOT EXISTS variant_frequency (
  id BIGSERIAL PRIMARY KEY,
  chr TEXT NOT NULL,
  pos BIGINT NOT NULL,
  ref TEXT NOT NULL,
  alt TEXT NOT NULL,
  case_count BIGINT NOT NULL DEFAULT 1,
  coord_hash BYTEA GENERATED ALWAYS AS (
    digest(
      int4send(octet_length(convert_to(chr, 'UTF8'))) || convert_to(chr, 'UTF8') ||
      int8send(pos) ||
      int4send(octet_length(convert_to(ref, 'UTF8'))) || convert_to(ref, 'UTF8') ||
      int4send(octet_length(convert_to(alt, 'UTF8'))) || convert_to(alt, 'UTF8'),
      'sha256'
    )
  ) STORED
);

-- Replace btree on (chr, pos, ref, alt, case_id) with btree on (coord_hash, case_id)
DROP INDEX IF EXISTS idx_variants_coord_case;
CREATE INDEX IF NOT EXISTS idx_variants_coord_hash_case ON variants(coord_hash, case_id);

-- Replace variant_frequency PK with surrogate + unique on coord_hash
CREATE UNIQUE INDEX IF NOT EXISTS variant_frequency_coord_hash_uniq ON variant_frequency(coord_hash);
```

All other indexes on `variants` (`idx_variants_case_type`, `idx_variants_case_gene`, `idx_variants_case_pos`, `idx_variants_case_consequence`, `idx_variants_case_func`, `idx_variants_search_document`) and the `update_variants_search_document` trigger are unchanged. They never indexed `ref`/`alt` directly so they have no row-size hazard.

### Application changes

**`src/main/storage/postgres/PostgresJsonImportRepository.ts` — `rebuildVariantFrequencyForCase`:**

```diff
 export async function rebuildVariantFrequencyForCase(
   client: Pick<PoolClient, 'query'>,
   schema: string,
   caseId: number
 ): Promise<void> {
   const schemaName = quoteIdentifier(schema)
   await client.query(
     `INSERT INTO ${schemaName}."variant_frequency" (chr, pos, ref, alt, case_count)
      SELECT chr, pos, ref, alt, 1
      FROM ${schemaName}."variants"
      WHERE case_id = $1
      GROUP BY chr, pos, ref, alt
-     ON CONFLICT (chr, pos, ref, alt)
+     ON CONFLICT (coord_hash)
      DO UPDATE SET case_count = ${schemaName}."variant_frequency".case_count + 1`,
     [caseId]
   )
 }
```

`coord_hash` is `GENERATED ALWAYS AS … STORED`, so the INSERT itself does not name it — postgres computes it during row construction. The `ON CONFLICT` targets the unique index on `coord_hash` directly.

**`src/main/storage/postgres/PostgresVariantReadRepository.ts` — variant list / detail JOIN:**

```diff
 const joins = [
-  `LEFT JOIN ${this.schemaName}."variant_frequency" vf ON vf.chr = v.chr AND vf.pos = v.pos AND vf.ref = v.ref AND vf.alt = v.alt`
+  `LEFT JOIN ${this.schemaName}."variant_frequency" vf ON vf.coord_hash = v.coord_hash`
 ]
```

Hash-only equality. Per Decision 4 we deliberately do *not* keep the column-equality predicates as a "defense-in-depth" layer — those would force postgres to detoast and byte-compare full `ref`/`alt` values on every JOINed row, which for long-read mega-alleles is a megabyte-scale compare per row. The cryptographic hash is the entire correctness contract.

### Hash input contract

The exact byte-encoding is fixed at the postgres layer (the `GENERATED ALWAYS AS` expression in Decision 2). No application code computes the hash for stored rows. Cross-case lookups that need to compute the hash for a specific `(chr, pos, ref, alt)` parameter set let postgres compute it inline using the same encoding:

```sql
WHERE v.coord_hash = digest(
  int4send(octet_length(convert_to($1::text, 'UTF8'))) || convert_to($1::text, 'UTF8') ||
  int8send($2::bigint) ||
  int4send(octet_length(convert_to($3::text, 'UTF8'))) || convert_to($3::text, 'UTF8') ||
  int4send(octet_length(convert_to($4::text, 'UTF8'))) || convert_to($4::text, 'UTF8'),
  'sha256'
)
```

The postgres planner caches this expression and the prepared-statement plan reuses it. The `digest()` call on the parameter side runs once per query, not per row. Lookup is *hash-only* (Decision 4) — no `chr/pos/ref/alt` column equality is appended.

For test fixtures, a JS reference implementation in `tests/main/storage/coord-hash-encoding.ts` (new helper module) reproduces the same byte layout using `Buffer.writeUInt32BE`/`Buffer.writeBigInt64BE` + `Buffer.from(text, 'utf8')` and a `node:crypto.createHash('sha256')` digest. The unit test asserts that the JS reference and the postgres-stored value match for a small panel of edge-case fixtures (empty-string fields are not allowed in VarLens but are tested for the encoding's injectivity; ASCII; multi-byte UTF-8 chr names; 9.7 KB allele).

## Data Flow

### INSERT (variants table) — large allele

1. Worker batch-inserts variants via `jsonb_to_recordset` (Phase 9 path).
2. PostgreSQL constructs each row, including auto-computing `coord_hash` from the generated expression. Computation cost: one sha256 over up to a few MB of allele text. Bound: O(allele_length).
3. PostgreSQL updates indexes:
   - `idx_variants_coord_hash_case`: 32-byte hash + 8-byte case_id ⇒ ~50-byte index entry. Always under the 2704-byte cap.
   - All other indexes (`idx_variants_case_*`): keys do not include ref/alt, so they were never affected by the cap.
4. Trigger `update_variants_search_document` fires; populates `search_document` (does not include ref/alt).
5. Row commits.

### UPSERT (variant_frequency rebuild) — large allele

1. After variants are committed, worker calls `rebuildVariantFrequencyForCase`.
2. The `INSERT … SELECT … GROUP BY chr, pos, ref, alt` produces one candidate row per unique coordinate tuple in the case.
3. PostgreSQL computes `coord_hash` for each candidate row (generated column).
4. `ON CONFLICT (coord_hash) DO UPDATE` matches against `variant_frequency_coord_hash_uniq`. Hit ⇒ increment `case_count`. Miss ⇒ insert.
5. Row commits.

### SELECT (cross-case coordinate lookup, e.g., `variant_frequency` JOIN)

1. Query plan: index seek on `idx_variants_coord_hash_case` for the `WHERE coord_hash = …` predicate.
2. Postgres reads the matching heap row.
3. Result returned. No column-equality defense layer — see Decision 4.

For huge alleles, postgres TOAST stores `ref`/`alt` out-of-line; the heap row carries an 18-byte pointer until a `SELECT` *actually projects* `ref` or `alt`. So:
- A list query that selects only `id, chr, pos, gene_symbol, …` (typical variant table view) never detoasts. Stays fast for long-read data.
- The variant detail panel's `SELECT v.*, vf.case_count` does detoast `ref`/`alt` once for the single matched row. That cost is unavoidable and was already paid pre-fix.
- Whichever queries select `ref`/`alt` in their projection are surfaced explicitly in the perf review of this phase; no new such query is introduced.

## Error Handling

- **`pgcrypto` extension missing.** `CREATE EXTENSION IF NOT EXISTS pgcrypto;` runs at init-db time. The container's postgres user must have permission to create extensions; for our docker dev image (postgres:15-alpine) the default `varlens` user is the schema owner and has this permission. CI's PG fixtures always init from these scripts, so a missing extension would surface immediately as a startup error — no runtime fallback needed. If we ship postgres GA later against a managed postgres where users can't `CREATE EXTENSION`, that's a separate ship-gate concern.
- **`coord_hash` not generated for some reason.** GENERATED ALWAYS AS … STORED is enforced by postgres at row insertion; you cannot supply an explicit value, and the column is `NOT NULL` because all inputs are `NOT NULL`. Test coverage: a unit test asserts that `coord_hash IS NOT NULL` after every insert path (single-file JSON, multi-file JSON, single-file VCF, multi-file VCF).
- **Hash collision (theoretical).** Both read and upsert paths rely on `coord_hash` equality alone. A read-path collision would surface a wrong-coordinate row (no app-level filtering protects against it); an upsert-path collision would merge two distinct variants' counts into one `variant_frequency` row. We deliberately do NOT add a column-equality WHERE on either side. A future reader who decides to "harden" the upsert with `ON CONFLICT (coord_hash) DO UPDATE … WHERE excluded.chr = …` would actually create a *worse* failure mode: postgres skips both update and insert when that WHERE clause evaluates false on a collision, silently dropping the row from `variant_frequency`. The same applies to read paths — adding column-equality predicates would make the planner detoast multi-MB allele values per JOINed row. The injective length-prefixed encoding (Decision 2) means false collisions can only come from sha256 itself, not input ambiguity; at our cardinality (≤ 10⁹ variants, birthday-bound P < 10⁻²⁰) this is well past cosmic-ray bit-flip territory.
- **Existing developer postgres dev database.** `make pg-reset && make pg-up` is the only path to pick up the schema change. The PR description and the PR comment for #179 will document this clearly. There is no automated detection of the old schema; running the new app code against the old schema will fail at the first `INSERT` into `variant_frequency` because the upsert references a `coord_hash` column that doesn't exist. The error message will be a plain postgres "column does not exist" — clear enough.

## Testing

Tests are organized to fail loudly on every dimension that matters. Test placement matches the existing VarLens convention: Vitest unit tests use a mocked `pg` client (no postgres infra), Playwright E2E tests gate on `VARLENS_RUN_POSTGRES_E2E=1` and run only after `make pg-reset && make pg-up`.

### Vitest unit tests (mocked client, no postgres infra)

- **`tests/main/storage/coord-hash-encoding.test.ts` (new).** Pure unit test for the JS reference implementation of the encoding. Asserts:
  - the byte layout of a known fixture matches the spec (4-byte BE length prefix, UTF-8 bytes, `int8send` BE encoding for `pos`),
  - the encoding is injective on three pairs of "almost-collision" inputs (e.g., `chr='1:2'`/`pos=3`/`ref='A'` vs `chr='1'`/`pos=2`/`ref='3:A'`),
  - sha256 of the encoded bytes is stable across runs.
  No postgres needed.
- **`tests/main/storage/postgres-json-import-repository.test.ts` (existing, extended).** Add an assertion that the `rebuildVariantFrequencyForCase` SQL emitted to the mocked client contains `ON CONFLICT (coord_hash)` (not `ON CONFLICT (chr, pos, ref, alt)`). No postgres needed.

### Playwright E2E (gated by `VARLENS_RUN_POSTGRES_E2E=1`, requires live postgres)

- **`tests/e2e/postgres-large-variant.e2e.ts` (new).** Mirrors the existing `tests/e2e/postgres-*.e2e.ts` gating pattern (`describe.skip(process.env.VARLENS_RUN_POSTGRES_E2E !== '1', …)`). Imports a small synthetic VCF fixture containing one HLA-style variant with a 9.7 KB ALT allele. Asserts:
  - the import completes without "index row size exceeds btree" error,
  - `SELECT coord_hash, length(coord_hash) FROM variants WHERE …` returns a 32-byte value,
  - that value matches the JS reference encoding (`coord-hash-encoding.ts`),
  - the cross-case JOIN against `variant_frequency` returns the expected `case_count` for the same row.
  This is the regression gate; runs in the same `VARLENS_RUN_POSTGRES_E2E=1` step as the other PG E2Es.
- **`tests/e2e/postgres-*.e2e.ts` (existing, no changes).** No-regression gate for non-large-allele paths. Runs unchanged.

### Perf benchmarks (gated)

- **`tests/perf/postgres-vcf-wgs-import.perf.test.ts` (existing).** This is the actual goal. Re-runs the GIAB HG002 fixture import against postgres. Pre-fix: aborts on the first HLA mega-allele. Post-fix: completes within budget (`VARLENS_PG_WGS_BUDGET_S`, default 600 s).
- **`tests/perf/sqlite-vcf-wgs-import.perf.test.ts` (existing, no schema change).** Re-run for fresh comparison; unchanged behavior expected.
- **`scripts/perf/compare-wgs-import.mjs` (existing).** Reads the latest pair of artifacts and writes a side-by-side comparison to `.planning/artifacts/perf/wgs-import/`. The artifact is the deliverable: it documents the post-fix PG/SQLite ratio and informs whether to escalate to `COPY FROM STDIN` (Phase 16 trigger).

### Verification gate

The phase is not done until:

1. `make typecheck` passes.
2. `make rebuild-node && make test` passes. Postgres infra is *not* required at this step — the new `coord-hash-encoding.test.ts` is a pure JS unit test, and the `postgres-json-import-repository.test.ts` extension keeps using the mocked `pg` client. The live-postgres large-allele test moved to E2E (gated separately).
3. `make pg-reset && make pg-up`, then `npm run rebuild:electron && make build`, then `VARLENS_RUN_POSTGRES_E2E=1 xvfb-run -a npx playwright test tests/e2e/postgres-*` passes (includes the new `postgres-large-variant.e2e.ts` and all existing PG E2E flows).
4. `npm run rebuild:electron` (perf tests need Electron's Node ABI).
5. `VARLENS_RUN_WGS_PERF=1 xvfb-run -a npx vitest run --project=perf tests/perf/postgres-vcf-wgs-import.perf.test.ts` passes within budget.
6. `VARLENS_RUN_WGS_PERF=1 xvfb-run -a npx vitest run --project=perf tests/perf/sqlite-vcf-wgs-import.perf.test.ts` passes (no SQLite regression).
7. `node scripts/perf/compare-wgs-import.mjs` writes a comparison artifact and the resulting PG/SQLite ratio is documented in the PR comment.
8. `make pg-down` cleans up.

Failures at any step block the phase. Lowering test or budget thresholds to make a failure pass is not acceptable (per AGENTS.md).

## Risks and Open Questions

- **`pgcrypto` permission in managed postgres.** Already addressed in non-goals: this phase ships against the docker dev image where extension creation is permitted. Managed-postgres GA will revisit.
- **TOAST detoast cost when `ref`/`alt` is projected.** When a query's `SELECT` list includes `ref` or `alt`, postgres detoasts those values. For multi-MB alleles this is non-trivial. Mitigation: with the hash-only JOIN (Decision 4), the JOIN itself never detoasts because it doesn't compare or read the `ref`/`alt` values. List queries that select `id, chr, pos, gene_symbol, …` stay fast. Detoast happens only in the variant detail panel where the projection actually selects `ref`/`alt` for one row at a time — the same cost as today, no regression. We will spot-check the perf benchmark output to confirm no list-query path was inadvertently rewritten to project ref/alt.
- **Hash function alignment with future GA4GH VRS.** If we ever ship VRS interop, we'd add a *separate* canonical-ID column with the VRS-spec hash and normalization. `coord_hash` is the lookup key; a future `vrs_id` column would be the interchange ID. Both can coexist.
- **Schema divergence between SQLite and postgres.** `coord_hash` exists only in postgres. The Kysely typed schema will have a postgres-specific extension. The repository abstraction already handles this kind of divergence (e.g., `cohort_variant_summary` is SQLite-only). Documented in non-goals; not a hidden risk.
