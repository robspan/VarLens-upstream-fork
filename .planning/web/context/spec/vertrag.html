§vertrag1 Charter and Target Picture

Charter. Make VarLens available as a web app on a self-hosted server, used internally by stakeholder users, instead of a desktop app with SQLite shuffling. The initial charter per the Task Profile is a minimal port with a manageable operating model. A cloud-native architecture is not in scope.

Target picture. The application speaks only to the Application Contract (§vertrag2) and is platform-agnostic. The dev and test mode with fake data is the primary working mode. Onboarding real data is a stakeholder decision outside this plan (§vertrag5).

Two stages, two plan books.

Stage Repository location Meaning
Stage 1: Concept konzept/ Minimum scope per the Task Profile. First viable web solution, manageable operating model. Aligned with the Task Profile.
Stage 2: Operations betrieb/ Full version beyond the minimum. Pilot maturity with OIDC, audit suitability, supply-chain assurance, Sovereign Cloud migration.

betrieb/ presupposes that konzept/ is complete. Each plan in betrieb/ states its concrete prerequisite at the top.

Bridge Clause. Every Stage 1 decision must keep the path to Stage 2 open. Concretely:

Stage 1 choice Bridge to Stage 2
SQLite as database Repository interface encapsulates engine differences; the SQLite-to-Postgres migration path is documented in code; user_id columns already exist
Argon2 authentication Auth sits behind an abstraction; OIDC arrives as a second implementation, not as a replacement
Single-server containers The application container is platform-agnostic (configured via ENV per §vertrag2); the compose stack only changes ENV values when switching cloud
Local state (OpenTofu) The module split is optional in Stage 1, but S3-API-compatible paths are not foreclosed
Basic restic backup Write and prune identities can be separated starting in Stage 1; Object Lock can be retrofitted on a fresh bucket
Caddy TLS Remains usable behind a central reverse proxy; the OIDC flow passes through Caddy
Audit-log table (already exists in VarLens) Append-only constraints are added in Stage 2 as a schema extension, not as a fresh table

Forbidden in Stage 1: anything that permanently rules out a Stage 2 capability (an S3 bucket without Object Lock preparation, hard-coded provider IDs without ENV indirection, auth wired directly into service code instead of behind an abstraction).

Bridge Clause checks belong in every Stage 1 phase Gate.

This plan is a Task Profile, not a complete specification. Concrete tool choices, ENV-variable names, endpoint schemas, compose files, and so on are produced at the start of the relevant phase, not in the plan.

§vertrag2 Application Contract

Defines the interface between the application and its runtime environment. Stable, maintained only with a version bump. Applies to both stages.

§vertrag2.1 Principles

Principle Meaning
ENV configuration No platform values in the repository
Stateless Persistent state only in the database
TLS outside Termination outside the application
Logs to stdout JSON, no file-based logs; /healthz as the health endpoint

§vertrag2.2 Platform-to-Application Interfaces

No What Guarantees Stage
1 Database via connection URL from ENV Consistency after commit; platform provides backups and engine updates both (see below)
2 OIDC Identity Provider via discovery URL from ENV PKCE flow, stable subject ID, MFA configurable Stage 2
3 Reverse proxy with TLS termination Terminates TLS, forwards HTTP, sets X-Forwarded-* both
4 Temporary write path from ENV Writable; at least container lifetime; factor 3 of the largest expected VCF file both

Interface 1 - stage-specific:

Stage DB engine Rationale
Stage 1 (Concept) SQLite (default) or PostgreSQL The Task Profile prefers SQLite. If multi-user operation is technically viable on SQLite, then SQLite. Otherwise Postgres. Assessment in konzept/bewertungen.html §bewertung2
Stage 2 (Operations) PostgreSQL RLS for multi-user isolation, append-only triggers for audit suitability, JSON operators. Not achievable with SQLite

In Stage 2, OIDC additionally requires Interface 2.

§vertrag2.3 Application Guarantees

Guarantee Detail
HTTP server Port from ENV, default 3000, HTTP only
/healthz 200 if database and migration are healthy, 503 otherwise; no login protection
JSON logs to stdout No file-based logs
Migrations on startup Fail fast, exit code != 0
ENV documentation in image label Which variables are expected
SemVer tag per image The platform pins explicitly
SIGTERM handler 30 s grace period

§vertrag2.4 Out of Scope

Backups, network access, monitoring, host hardening, concrete tool choices, application mail, object storage. VCF data lives in the database; region files and gene reference data are database blobs or build-time assets.

§vertrag2.5 Versioning

SemVer. Major = breaking, Minor = backwards-compatible extension, Patch = clarification. Contract changes require an ADR. The contract and the plan are versioned independently.

§vertrag5 Real Data - outside this plan

AWS Sovereign Cloud + internal stakeholder use = standard clinic IT (relevant from Stage 2). Sovereignty, the data-processing-agreement path, and §203 compatibility are addressed by design. Compliance documentation (DPIA, TOM, authorisation and deletion concepts) is run via stakeholder routine. The plan delivers technical input on request but does not drive the compliance documentation apparatus.

Self-binding: no real data in Stage 1. Real data only after the migration block onto AWS Sovereign Cloud, and even then only as a stakeholder decision.
