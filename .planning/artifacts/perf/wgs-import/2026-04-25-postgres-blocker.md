# WGS import perf — postgres (BLOCKED, schema limitation)

- timestamp: 2026-04-25
- fixture: tests/.cache/wgs/HG002_GRCh38_1_22_v4.2.1_benchmark.vcf.gz
- backend: postgres (15-alpine, varlens-postgres-dev)
- elapsed: N/A (import aborted)

## Failure

```
[PostgresImportExecutor] worker error:
  index row size 3600 exceeds btree version 4 maximum 2704 for index "idx_variants_coord_case"
```

The Phase 6 `idx_variants_coord_case` btree index covers
`(chr, pos, ref, alt, case_id)`. PostgreSQL's btree page max is 2704
bytes per row. Five GIAB HG002 v4.2.1 variants on chr6 (HLA region,
positions 31042318 / 31311231 / 31428987 / 31453720 / 32648270) exceed
this with very long REF or ALT strings (up to 9.7 KB each):

| line     | bytes | ref | alt |  chr | pos      |
|----------|-------|-----|-----|------|----------|
| 1552802  | 7373  |   1 |7372 | chr6 | 31042318 |
| 1555721  | 9705  |9704 |   1 | chr6 | 31311231 |
| 1558664  | 3176  |   1 |3175 | chr6 | 31428987 |
| 1558938  | 2891  |2890 |   1 | chr6 | 31453720 |
| 1565008  | 2923  |2922 |   1 | chr6 | 32648270 |

SQLite's b-tree has no equivalent per-row size limit, so the SQLite
WGS perf benchmark (sibling artifact this directory) imports the same
fixture cleanly in ~52 seconds.

## Escalation

This is a Phase 6 schema limitation surfaced by Phase 9's WGS perf
gate. It is **NOT a Phase 9 worker bug** — the same import failure
would occur with the pre-Phase-9 main-thread executor.

Recommended follow-up: a Phase 6.x or Phase 9.1 schema-fix phase that
either:

1. Switches `idx_variants_coord_case` to use an expression index over
   `md5(chr || ':' || pos || ':' || ref || ':' || alt)` (cheap hash
   index, lookup-equivalent for exact-match queries that the variants
   page uses).
2. Replaces the index with a partial index on rows where
   `length(ref) + length(alt) < 2700` and falls back to seq-scan for
   the few large variants.
3. Moves `ref`/`alt` for large variants to a side table keyed by
   variant_id, with a sentinel in the main table.

A pre-filtered WGS perf fixture (variants with `length(ref)+length(alt)
< 2700`) would let us measure PG vs SQLite ratio on the
small-variant-only subset; that would be a useful interim measurement
once the Phase 6.x schema fix lands.

The escalation rule from Phase 9 spec ("postgres > 2× sqlite → COPY
FROM STDIN follow-up phase") does not apply here — this is a
correctness gate, not a perf delta. The perf escalation can only be
evaluated once the schema gate is unblocked.

## Resolution

Resolved by Phase 9.1: switched postgres `idx_variants_coord_case` and
`variant_frequency.PRIMARY KEY` to sha256-keyed indexes via a stored
generated `coord_hash BYTEA` column. The cryptographic hash is bounded
at 32 bytes regardless of allele size, scaling cleanly to long-read
multi-MB alleles.

The first post-fix WGS run also surfaced an unrelated memory issue
(single-transaction shape OOMing the worker on multi-million-variant
files). Fixed in the same phase by switching the postgres-import-worker
to commit per batch — see commit `perf(workers): commit per batch in
postgres import worker`. The benchmark now runs with a 1-GB Node heap
cap and ~417-MB postgres backend peak, fitting on 8-GB-RAM machines.
